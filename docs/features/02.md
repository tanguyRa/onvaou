## Goal

Define the canonical PostGIS schema for the `events` and `source_hashes` tables and integrate them into the existing golang-migrate migration system so the database can evolve without data loss.

## Scope

### Migration files

Add two new migration files to the existing `db/migrations/` directory (continuing the existing numbering):

`**000005_postgis_events.up.sql**`

- `CREATE EXTENSION IF NOT EXISTS postgis;`
- Create the `events` table with all fields from the architecture spec, using `GEOGRAPHY(Point, 4326)` for the `geom` column and `gen_random_uuid()` for `event_id`
- `CREATE INDEX idx_events_geom ON events USING GIST(geom);`
- `CREATE INDEX idx_events_start_dt ON events (start_dt);`
- Create the `source_hashes` table: `(source_tag TEXT, content_hash TEXT, event_id UUID REFERENCES events(event_id) ON DELETE CASCADE, PRIMARY KEY (source_tag, content_hash))`

`**000005_postgis_events.down.sql**`

- Drop `source_hashes`, `events`, and the PostGIS extension in reverse order

### PostGIS extension

The `postgis` extension must be enabled in the Postgres container. Update `compose.yml` to use `postgis/postgis:16-3.4-alpine` as the Postgres image instead of `postgres:16-alpine` — this image ships with PostGIS pre-installed.

### sqlc configuration

Add basic sqlc queries for the `events` table in `db/queries/events.sql` so the Go backend can read event counts or featured events if needed in future. The Python service uses raw `asyncpg` SQL, not sqlc.

## Acceptance Criteria

- [ ] `postgis/postgis:16-3.4-alpine` image used in `compose.yml`
- [ ] Running `migrate -path db/migrations -database $DATABASE_URL up` applies migration 000005 cleanly
- [ ] `events` table has correct columns including `geom GEOGRAPHY(Point, 4326)`
- [ ] GiST index `idx_events_geom` exists and is used by `ST_DWithin` queries (verify with `EXPLAIN ANALYZE`)
- [ ] `source_hashes` table exists with correct FK constraint
- [ ] Down migration cleanly removes all new objects
